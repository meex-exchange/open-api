Basic information

API Basic information

This article lists the REST interface's baseurl https://openapi.meex.com

All interfaces will return a JSON object or array.

If there is an array in the response, the array elements are arranged in reverse chronological order, and the earlier the data is, the earlier it is.

All times and timestamps are in UNIX time, measured in milliseconds.

HTTPreturn code

HTTP 4XX Error codes are used to indicate incorrect request content, behavior, and format.

HTTP 429 The error code indicates that the warning access frequency has exceeded the limit and the IP address is about to be banned.

HTTP 418 Said to continue accessing after receiving 429, so it was banned.

HTTP 5XX The returned error code is an internal system error; This indicates that the problem is on the server side.

When dealing with this error, never treat it as a failed task, as the execution status is unknown, and it can be either a success or a failure.

HTTP 504 Indicates that the API server has submitted a request to the business core but failed to obtain a response，

It is particularly important to note that code504does not represent a request failure, but rather an unknown。

It is highly likely that the execution has already been completed, or it may have failed and further confirmation is needed.

Any interface may return anERROR; The incorrect return payload is as follows:

{
    "retCode":1121,
    "retMsg":"Invalid symbol."
    "data":{}
}

Access Restrictions

There will be a limited frequency explanation below each interface.

Violating frequency restrictions will result in receiving HTTP 429, which is a warning.

When receiving the 429 alarm, the caller should reduce the access frequency or stop accessing.

All interfaces accessed 300 times/s

Interface authentication type

  • Each interface has its own authentication type, which determines what authentication should be performed during access
  • If an API-key is required, it should be passed in the HTTP header as an X-CH-APIKEY field
  • API-key and API-secret are case sensitive
  • You can modify the permissions of API-key in the web user center.
  • For example, reading account information, sending transaction instructions, and sending withdrawal instructions

Interface that requires signature (TRADE and USER_DATA)

  • Calling TRADE or USER_DATAinterface, signature parameters should be passed in the HTTP header as X-CH-SIGN fields
  • The signature uses the HMAC SHA256 algorithm . The API-Secret corresponding to API-KEY serves as the key for HMAC SHA256
  • The request header of X-CH-SIGNtakes timestamp+method+requestPath+body string (+represents string connection) as the operation object
  • The timestamp value is the same as the X-CH-TS request header, with method being the request method and all uppercase letters: GET/POST
  • RequestPath is the request interface path, for example:/sapi/v1/order? orderId=211222334&symbol=btcusdt
  • Body is the string of the requested body (post only). If it is a GET request, the body can be omitted
  • Signature case insensitive.

Time synchronization security

  • The signing interface needs to pass a timestamp in the HTTP header using the X-CH-TSfield, which should be the Unix timestamp (ms) at the time the request was sent. e.g. 1528394129373
  • When the server receives a request, it will determine the timestamp in the request. If it was sent 5000 ms before, the request will be considered invalid. This time window value can be customized by sending the optional parameter recvWindow.
  • In addition, if the server calculates that the client timestamp is more than one second in the 'future' of the server time, the request will also be rejected.
  • Logical pseudocode:
if (timestamp < (serverTime + 1000) && (serverTime - timestamp) <= recvWindow) {
  // process request
} else {
  // reject request
}

Regarding transaction timeliness

The internet situation is not 100% reliable and can not be completely relied on, so the delay from your program's local connection to the exchange server may fluctuate This is the purpose of setting recvWindow If you are engaged in high-frequency trading and have high requirements for transaction timeliness, you can flexibly set recvWindow to meet your requirements. It is not recommended to use recvWindow for more than 5 seconds

Signature

Sample

/**
 * Generate sign
 **/
private String toSign(String timestamp, String method, String requestPath,
                             String queryString, String body, String secretKey)
                             throws Exception {
    // Signature string
    String preHash = preHash(timestamp, method
    , requestPath, queryString, body);
    byte[] secretKeyBytes = secretKey.getBytes("UTF-8");
    SecretKeySpec secretKeySpec =
    new SecretKeySpec(secretKeyBytes, "HmacSHA256");
    Mac mac = (Mac) MAC.clone();
    mac.init(secretKeySpec);
    return Hex.encodeHexString(mac.doFinal(preHash.getBytes("UTF-8")));
}

/**
 * Signature string
 **/
private String preHash(String timestamp, String method, String requestPath,
String queryString, String body) {

    StringBuilder preHash = new StringBuilder();
    preHash.append(timestamp);
    preHash.append(method.toUpperCase());
    preHash.append(requestPath);
    if (org.apache.commons.lang3.StringUtils.isNotEmpty(queryString)) {
        preHash.append("?").append(queryString);
    }
    if (org.apache.commons.lang3.StringUtils.isNotEmpty(body)) {
        preHash.append(body);
    }
    return preHash.toString();
}

/**
 * queryString
 **/
private String queryString(ServerHttpRequest request) {
    String url = request.getURI().toString();
    String queryString = "";
    if (url.contains("?")) {
        queryString = url.substring(url.lastIndexOf("?") + 1);
    }
    return queryString;
}
